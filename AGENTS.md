# Правила работы AI-агента (AGENTS.md)

Эти правила являются строгими и обязательными для исполнения любым AI-агентом, работающим с проектом GotMeds.

## Правило 1: Русский язык по умолчанию
* Всё обсуждение проекта, планирование и ответы на запросы должны вестись **на русском языке**.
* **Комментарии в коде** и любая генерируемая документация должны быть написаны **на русском языке**.
* *Исключение:* Синтаксис кода (названия переменных, функций, классов, базы данных и т.д.) пишется на английском языке в соответствии с общепринятыми стандартами программирования.

## Правило 2: Строгое следование спецификациям (SDD)
* Разработка ведется **исключительно** на основе утвержденных спецификаций (находящихся в папке `spec/`).
* В проекте **не должно быть** функционала, логики или архитектурных решений, которые не описаны в спецификациях. 
* Любые инициативы или внедрение сторонних фич "от себя" или "извне" строго запрещены.
* **СТОП-ПРАВИЛО:** Если пользователь просит реализовать что-то, чего нет в текущих спецификациях, агент **ОБЯЗАН остановиться**. Агент должен предупредить пользователя об отсутствии этого пункта в спецификации и **не выполнять задачу** до тех пор, пока спецификация не будет официально обновлена пользователем.

## Правило 3: Интеграция с Agent Skills
* Данный файл (`AGENTS.md`) является главным документом с правилами для ИИ.
* Для технического исполнения этих правил (в частности, автоматического поиска спецификаций) проект использует Agent Skill.
* Скилл находится по пути: `.gemini/skills/gotmeds-spec-reader/SKILL.md`. Любой совместимый агент должен использовать этот скилл для ускорения навигации по папке `spec/` и контроля исполнения правил.

## Правило 4: Строгий Git Workflow (Работа с ветками и PR)
* **Создание ветки:** Каждый раз, когда создается или берется в работу новый план (этап/фича), агент обязан создать и переключиться на новую ветку для разработки этого плана.
* **Без автоматической публикации:** После завершения разработки агент не должен автоматически делать `commit`, `git push` или создавать Pull Request без прямой просьбы пользователя.
* **Завершение и Push:** Когда пользователь просит завершить работу по плану или сделать `push`, агент обязан проверить состояние рабочей копии, перенести текущий план из `plans/active/` в `plans/completed/`, обновить дорожную карту, затем сделать `commit` и отправить изменения на GitHub (`git push`).
* **Pull Request:** Сразу после пуша, агент должен инициировать создание Pull Request. Для этого агент использует команду `gh pr create` (если установлен GitHub CLI) **ИЛИ** выводит в чат прямую ссылку на создание PR в браузере. (Слияние / merge выполняет сам пользователь на GitHub).
* **Синхронизация перед новой задачей:** Если Pull Request уже создан и пользователь просит продолжить работу или начать следующий план, агент обязан сначала убедиться, что рабочая копия не содержит незакоммиченных изменений, которые нельзя безопасно оставить, затем переключиться на `main` (`git checkout main`), подтянуть свежие изменения (`git pull`) и только после этого организовывать новый план и создавать новую рабочую ветку.
* **Без самостоятельного merge:** Агент не выполняет merge Pull Request самостоятельно. Merge выполняет владелец проекта вручную на GitHub.

## Правило 5: Управление планами разработки и дорожной картой (Active & Completed Plans)
* В корне проекта существует директория `plans/`, состоящая из двух папок: `plans/active/` и `plans/completed/`, а также директория `roadmap/` для хронологической истории завершенных работ.
* **Создание плана:** Любой новый план разработки должен быть сначала сохранен как отдельный markdown-файл в папке `plans/active/`.
* **Разработка:** Пока функционал по плану не будет полностью реализован и протестирован пользователем, план находится в папке `active`.
* **Завершение и перенос (До коммита):** Когда пользователь протестировал функционал и дает команду на пуш (`push`), агент **ОБЯЗАН** сначала перенести файл текущего плана из `plans/active/` в `plans/completed/` (для сохранения истории).
* **Ведение дорожной карты:** Сразу после переноса плана в `completed`, агент обязан обновить `roadmap/chronology.md` и все затронутые файлы областей в `roadmap/areas/`. Только после этого выполняется `git add`, `commit` и `push`.
* **Новые области:** Если завершенный план затронул область, которой еще нет в дорожной карте, агент обязан создать новый файл области в `roadmap/areas/` и добавить ссылку на него в `roadmap/README.md`.
* **Хронологический порядок:** Новые записи добавляются в конец `roadmap/chronology.md` и соответствующих файлов областей, чтобы порядок шел от старых работ к новым.
* **Ссылка на план:** Каждая запись дорожной карты должна ссылаться на файл завершенного плана из `plans/completed/`.
* **Pull Request в дорожной карте:** Если Pull Request еще не создан на момент обновления дорожной карты, в поле `Pull request` нужно указывать `будет создан после push`. После создания Pull Request агент не обязан делать отдельный commit только ради добавления номера или ссылки Pull Request, если пользователь явно не попросил обновить эту запись.
* **Защита чужих изменений:** Если в рабочей копии есть незакоммиченные изменения, которые не относятся к текущему плану, агент не должен перезаписывать, переносить или откатывать их без явного разрешения пользователя. В такой ситуации нужно явно сообщить о состоянии рабочей копии и выбрать безопасное продолжение.

## Правило 6: Шаблон плана и обязательная пользовательская проверка
* Каждый план разработки **обязан** содержать раздел **«Пользовательская проверка (Пошаговый чеклист)»**.
* Этот раздел описывает **конкретные шаги**, которые пользователь выполняет для ручной проверки результата. Для каждой задачи указывается:
  1. Что нужно сделать (команда, URL, действие).
  2. ✅ Ожидаемый результат (что пользователь должен увидеть).
  3. ❌ Что делать, если не работает (шаг устранения).
* В конце раздела — **итоговый чеклист** с галочками для финальной приёмки.
* Шаблон плана находится в файле: `plans/TEMPLATE.md`. Все новые планы **должны** создаваться на его основе.
