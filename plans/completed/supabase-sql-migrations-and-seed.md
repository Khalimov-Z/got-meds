# План разработки: Supabase SQL migrations and seed

**Статус:** Завершено (Completed)
**Основание:** [spec/technical/supabase-platform-migration.md](../../spec/technical/supabase-platform-migration.md), раздел 6.2; [spec/features/data-architecture/data-architecture.md](../../spec/features/data-architecture/data-architecture.md), раздел 3.1; [spec/technical/security-performance.md](../../spec/technical/security-performance.md)

## Цель
Перенести источник воспроизведения схемы и тестовых данных из Prisma migrations/seed в Supabase SQL-подход, не меняя пользовательское поведение, runtime data layer, UI, авторизацию и доменную модель GotMeds.

По завершении этапа новая чистая Supabase PostgreSQL база должна воспроизводиться SQL-миграциями и SQL-seed без потери текущей структуры: enum, таблиц, foreign keys, уникальных ограничений, индексов, `pg_trgm` и тестового набора данных.

## Границы этапа
- Входит: создание SQL-миграций Supabase, перенос текущей структуры из `prisma/schema.prisma` и существующих Prisma migrations, перенос seed-данных в SQL, проверка на чистой тестовой Supabase PostgreSQL базе.
- Входит: аудит расхождения между текущей Prisma-схемой и спецификацией данных, включая таблицу жалоб `Pharmacy_Reports`; если для полного соответствия потребуется расширение схемы сверх текущей Prisma-схемы, план должен быть обновлен и повторно одобрен до реализации такого изменения.
- Не входит: удаление Prisma, изменение runtime-кода приложения, переход на Supabase SDK, RPC/functions, Supabase Auth, RLS-политики, Storage, Realtime, изменение UI, изменение production-данных и автоматический деплой.
- Prisma migrations и `prisma/seed.ts` остаются в репозитории как переходный rollback-слой до отдельного финального плана удаления Prisma.
- Секреты Supabase, connection string и значения `DATABASE_URL` не коммитятся в репозиторий.

## Пользовательская проверка (Пошаговый чеклист)

> Выполните каждый шаг по порядку. Если на любом шаге результат не совпадает с ожидаемым — значит что-то пошло не так.

### Проверка задачи 1: SQL-артефакты созданы
1. Откройте `supabase/migrations/`.
2. Откройте `supabase/seed.sql`.
3. Откройте `supabase/README.md`.
4. ✅ **Ожидаемый результат:** есть SQL-миграция схемы, SQL-seed и инструкция проверки; файлы не содержат реальных Supabase-секретов или production-данных.
5. ❌ **Если не работает:** остановитесь и верните план на доработку; секреты нужно удалить до любых дальнейших проверок.

### Проверка задачи 2: Чистая Supabase база воспроизводится
1. Создайте отдельную пустую тестовую Supabase PostgreSQL базу или проект, не production.
2. Примените SQL-миграцию из `supabase/migrations/` через Supabase SQL Editor, `psql` или Supabase CLI.
3. Примените `supabase/seed.sql`.
4. ✅ **Ожидаемый результат:** миграция и seed выполняются без SQL-ошибок на пустой базе.
5. ❌ **Если не работает:** сохраните текст ошибки и не применяйте SQL к production-базе; rollback — продолжать использовать Prisma migrations до исправления SQL.

### Проверка задачи 3: Структура и индексы совпадают с ожиданием
1. В Supabase SQL Editor выполните проверку `select extname from pg_extension where extname = 'pg_trgm';`.
2. Проверьте наличие таблиц `cities`, `admins`, `pharmacies`, `products`, `inventory`, `product_aliases`, `unmapped_strings`, `search_logs`.
3. Проверьте GIN-индексы `idx_products_name_trgm` и `idx_product_aliases_original_string_trgm`.
4. ✅ **Ожидаемый результат:** `pg_trgm` включен, таблицы созданы, индексы fuzzy search существуют.
5. ❌ **Если не работает:** остановитесь; SQL-миграция не считается готовой до восстановления структуры и индексов.

### Проверка задачи 4: Seed-данные загружены
1. В Supabase SQL Editor проверьте счетчики таблиц.
2. Ожидаемые счетчики: `cities=1`, `pharmacies=3`, `products=16`, `product_aliases=5`, `inventory=24`, `admins=1`.
3. Проверьте, что есть товары `Нурофен`, `Диазепам`, `Но-Шпа`, `Дротаверин`, а `Диазепам` имеет `is_social_risk = true`.
4. ✅ **Ожидаемый результат:** тестовый набор соответствует текущему `prisma/seed.ts` по составу данных.
5. ❌ **Если не работает:** остановитесь; seed нужно исправить до проверки приложения.

### Проверка задачи 5: Приложение работает с SQL-созданной базой
1. Локально укажите `DATABASE_URL` на тестовую Supabase базу, созданную SQL-миграциями.
2. Выполните `npx prisma generate`.
3. Запустите приложение командой `npm run dev`.
4. Проверьте поиск `нурофен`, алиас `Нуроф таб 200мг`, restricted-запрос `диазепам`, страницу `Но-Шпа` и карту аптек.
5. ✅ **Ожидаемый результат:** текущий Prisma runtime работает с базой, созданной SQL-миграциями, без изменения поведения приложения.
6. ❌ **Если не работает:** верните `DATABASE_URL` на прежнюю рабочую базу и приложите ошибку runtime-проверки.

### Проверка задачи 6: Сборка и качество
1. Выполните `npm run lint`.
2. Выполните `npm run build`.
3. Выполните `git diff --check`.
4. ✅ **Ожидаемый результат:** lint, build и проверка diff проходят успешно.
5. ❌ **Если не работает:** этап не считается завершенным; исправьте ошибку и повторите проверки.

### Итоговый критерий: Этап завершён, если
- [x] Чистая Supabase PostgreSQL база воспроизводится SQL-миграцией.
- [x] SQL-seed загружает тестовый набор, эквивалентный текущему `prisma/seed.ts`.
