# План разработки: Этап 3 (Базовый интерфейс и интеграция поиска)

**Статус:** Завершён (Completed)
**Основание:** [roadmap.md](../../spec/architecture/roadmap.md), [client-app-spec.md](../../spec/features/client-app/client-app-spec.md), [design-system.md](../../spec/architecture/design-system.md), [search-engine.md](../../spec/features/search-engine/search-engine.md), [api-spec.md](../../spec/architecture/api-spec.md)

## Цель
Реализовать базовый пользовательский интерфейс поиска на главной странице GotMeds и подключить его к уже готовому поисковому API этапа 2. Пользователь должен открыть `/`, ввести название препарата, увидеть подсказки/карточки результатов и получить понятное состояние при пустой выдаче.

## Задачи:
- [x] **1. Дизайн-система и базовые стили:**
  - Настроить `src/app/globals.css` на основе HSL-токенов из дизайн-системы.
  - Реализовать mobile-first контейнер приложения с расширенной desktop-компоновкой главной страницы.
  - Усилить визуальную иерархию главной страницы: поисковая зона должна выглядеть как главный рабочий инструмент, а не как обычная форма в базовой карточке.
  - Добавить состояния hover/focus и плавные CSS-переходы 0.2-0.3s для интерактивных элементов.
  - Подготовить стили для skeleton-состояний без сторонних UI-библиотек.

- [x] **2. Главная страница поиска (`/`):**
  - Заменить заглушку этапа 1 на стартовый экран поисковика.
  - Показать логотип GotMeds, приветственный текст «Найди лекарство в Гудермесе», вторичный текст о поиске препаратов, витаминов и медтехники и крупную строку поиска.
  - Добавить ссылки/кнопки популярных категорий: «Медтехника», «Витамины», «Мать и дитя».
  - Добавить обучающий блок «Как это работает?» согласно спецификации клиентского приложения.
  - Добавить юридический дисклеймер: «Сервис носит информационный характер. Имеются противопоказания. Не является публичной офертой».

- [x] **3. Интеграция с поисковым API:**
  - Создать клиентский компонент поиска для ввода запроса.
  - Вызывать `GET /api/search?q=<запрос>` при вводе пользователя с небольшим debounce.
  - Показывать skeleton-состояние во время запроса.
  - Отображать результаты в виде карточек препарата с названием, категорией, дозировкой/ценой при наличии и признаком рецептурности.
  - Ограничить UI существующим контрактом этапа 2, не добавляя неописанные API-поля.

- [x] **4. Состояния результата:**
  - Для успешной выдачи показывать список карточек с сортировкой, пришедшей от API.
  - Для пустой выдачи показывать сообщение «Препарат не найден. Возможно, он снят с производства или мы еще не добавили его в базу.»
  - Для ошибки API показывать короткое сообщение о невозможности выполнить поиск.
  - Для пустой строки не выполнять сетевой запрос.

## Пользовательская проверка (Пошаговый чеклист)

> Выполните каждый шаг по порядку. Если на любом шаге результат не совпадает с ожидаемым — значит что-то пошло не так.

### Проверка задачи 1: Базовая страница и стили
1. Откройте терминал в папке проекта.
2. Запустите приложение командой: `npm run dev`
3. Откройте в браузере: **http://localhost:3000**
4. ✅ **Ожидаемый результат:** Видна главная страница GotMeds с крупной строкой поиска, аккуратной мобильной колонкой, категориями, блоком «Как это работает?» и юридическим дисклеймером.
5. ❌ **Если не работает:** Проверьте вывод `npm run dev`; если есть ошибка сборки, исправьте файл, указанный в терминале.

### Проверка задачи 2: Поиск с опечаткой
1. На странице **http://localhost:3000** введите в поиск: `нурафен`
2. ✅ **Ожидаемый результат:** Под строкой поиска появляется карточка «Нурофен», найденная через Fuzzy Search.
3. ❌ **Если пусто:** Проверьте, что этап 2 применён, база заполнена seed-данными и эндпоинт **http://localhost:3000/api/search?q=нурафен** возвращает JSON с результатом.

### Проверка задачи 3: Поиск по точному названию
1. Введите в поиск: `Парацетамол`
2. ✅ **Ожидаемый результат:** В выдаче отображается карточка «Парацетамол» с базовой информацией из поискового API.
3. ❌ **Если пусто:** Проверьте подключение к базе данных и результат эндпоинта **http://localhost:3000/api/search?q=Парацетамол**.

### Проверка задачи 4: Пустая выдача
1. Введите в поиск: `хххххх`
2. ✅ **Ожидаемый результат:** Отображается сообщение «Препарат не найден. Возможно, он снят с производства или мы еще не добавили его в базу.»
3. ❌ **Если отображаются случайные препараты:** Проверьте порог схожести в `searchProducts(query)`.

### Проверка задачи 5: Ошибка API
1. Временно остановите базу данных или укажите неверный `DATABASE_URL`.
2. Введите любой поисковый запрос.
3. ✅ **Ожидаемый результат:** Интерфейс показывает понятное сообщение о невозможности выполнить поиск, страница не ломается.
4. ❌ **Если страница падает:** Добавьте обработку ошибки вокруг вызова `/api/search`.

### Проверка задачи 6: Desktop-компоновка главной страницы
1. Откройте **http://localhost:3000** в браузере на экране шириной от `900px`.
2. Убедитесь, что окно браузера не находится в режиме мобильной эмуляции.
3. ✅ **Ожидаемый результат:** Главная страница использует широкую desktop-компоновку: поисковая зона является главным визуальным элементом, контент не зажат в мобильную колонку, а блок «Как это работает?» разложен в несколько колонок.
4. ❌ **Если страница выглядит как узкая мобильная колонка:** Проверьте актуальность `spec/features/client-app/client-app-spec.md`, `spec/architecture/design-system.md` и media query в `src/components/search-experience.module.css`.

### Итоговый критерий: Этап 3 завершён, если
- [x] Главная страница `/` заменена с заглушки на поисковый интерфейс.
- [x] Визуальные стили соответствуют дизайн-системе из `spec/architecture/design-system.md`.
- [x] На desktop-экранах главная страница использует широкую адаптивную компоновку, а не узкую мобильную колонку.
- [x] Поиск вызывает `/api/search` и показывает карточки результатов.
- [x] Запрос `нурафен` находит «Нурофен».
- [x] Запрос `Парацетамол` показывает точное совпадение.
- [x] Запрос `хххххх` показывает пустое состояние без случайных совпадений.
- [x] Ошибка API обрабатывается без падения страницы.
