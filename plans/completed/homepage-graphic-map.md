# План разработки: Графическая карта главной страницы GotMeds

**Статус:** Завершён (Completed)
**Основание:** [design-system.md](../../spec/technical/design-system.md), [client-app-spec.md](../../spec/features/client-app/client-app-spec.md)

## Цель
Создать отдельный дизайн-справочник для будущего редизайна главной страницы GotMeds: разобрать дизайнерские desktop и mobile-макеты на визуальные токены, компоненты, секции, responsive-правила и карту будущего переноса в рабочий интерфейс.

План не меняет текущую продуктовую логику и не подключает новую страницу к пользовательским маршрутам MVP. HTML-страница графической карты должна быть внутренним справочным артефактом для разработки дизайна.

## Зафиксированные решения
- **Bottom navigation:** mobile-only дизайнерский ориентир. В текущем MVP не реализуется как функциональная навигация; пункты вроде "Избранное" и "Справка" не считаются обязательством реализации без отдельной спецификации.
- **Stats bar:** mobile-only визуальный блок доверия. Используется в графической карте без привязки к реальным метрикам и без обязательства показывать его на desktop.
- **Search behavior:** desktop и mobile в рамках дизайн-карты показывают поиск на месте, без автоматического перехода на карту. Любое изменение реальной продуктовой логики поиска требует отдельного согласования со спецификацией.
- **Mobile categories:** категории не скрываются полностью; для mobile нужно показать альтернативный компактный паттерн, предпочтительно горизонтальные chips.
- **Hero image:** использовать новое похожее изображение высокого качества: светлый интерьер аптеки, мягкий фокус, голубовато-белая палитра, чистая медицинская атмосфера.

  - Отдельно отметить элементы, которые требуют обновления спецификации перед реальной реализацией.

- [x] **7. Проверка качества:**

## Пользовательская проверка (Пошаговый чеклист)

> Выполните каждый шаг по порядку. Если на любом шаге результат не совпадает с ожидаемым — значит что-то пошло не так.

### Проверка задачи 1: Наличие графической карты
1. Откройте HTML-страницу графической карты в браузере.
2. Просмотрите оглавление и основные секции страницы.
3. ✅ **Ожидаемый результат:** На странице есть разделы про макеты, токены, компоненты, responsive-правила и карту будущего переноса.
4. ❌ **Если не работает:** Проверить путь к HTML-файлу и наличие всех секций в разметке.

### Проверка задачи 2: Desktop-разбор
1. Найдите на странице блок desktop preview.
2. Сверьте его с дизайнерским desktop-макетом.
3. ✅ **Ожидаемый результат:** В preview отражены header, hero, поисковая glass-панель, категории, декоративная медицинская графика, секция "Как это работает?" и footer.
4. ❌ **Если не работает:** Дополнить desktop preview недостающими элементами и проверить responsive-стили.

### Проверка задачи 3: Mobile-разбор
1. Найдите на странице блок mobile preview.
2. Сверьте его с дизайнерским mobile-макетом.
3. ✅ **Ожидаемый результат:** В preview отражены compact header, центрированный hero, mobile search panel, альтернативные категории, horizontal steps, stats bar и bottom navigation как визуальные ориентиры.
4. ❌ **Если не работает:** Проверить ширину mobile preview, overflow для горизонтальных элементов и состав mobile-only блоков.

### Проверка задачи 4: Зафиксированные решения
1. Найдите раздел с принятыми решениями.
2. Проверьте пункты про bottom navigation, stats bar, поиск, mobile categories и hero image.
3. ✅ **Ожидаемый результат:** Все решения из плана отображены на странице и не трактуются как обязательная реализация MVP.
4. ❌ **Если не работает:** Добавить явные пометки о статусе каждого спорного элемента.

### Проверка задачи 5: Изоляция от текущего приложения
1. Откройте текущую главную страницу приложения.
2. Проверьте, что она не изменилась из-за графической карты.
3. ✅ **Ожидаемый результат:** Рабочая главная страница и текущие маршруты приложения не изменены.
4. ❌ **Если не работает:** Проверить, что HTML-страница графической карты не подключена к production-роутам и не меняет общие стили приложения.
