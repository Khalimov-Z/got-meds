# План разработки: Этап 7 (Аналитика нулевой выдачи)

**Статус:** Завершён (Completed)
**Основание:** [implementation-plan.md](../../spec/technical/implementation-plan.md), [api-spec.md](../../spec/technical/api-spec.md), [data-architecture.md](../../spec/features/data-architecture/data-architecture.md), [search-engine.md](../../spec/features/search-engine/search-engine.md), [06-additional-features.md](../../spec/user-stories/06-additional-features.md), [admin-panel-spec.md](../../spec/features/admin-panel/admin-panel-spec.md)

## Цель
Реализовать базовый сбор аналитики неудовлетворенного спроса: если пользователь ищет препарат и система не находит результат, запрос незаметно записывается в `Search_Logs` для будущего B2B-дашборда админки.

## Задачи:
- [x] **1. Таблица `Search_Logs`:**
  - Добавить модель `SearchLog` в Prisma согласно спецификации данных.
  - Связать лог с городом через `city_id`.
  - Создать миграцию для таблицы `search_logs`.

- [x] **2. Server Action логирования:**
  - Добавить `logZeroResultSearch(searchTerm: string, cityId: string, lat?: number, lng?: number)`.
  - Валидировать пустой запрос и несуществующий город.
  - Записывать `search_term`, `city_id`, координаты при наличии, `results_count = 0` и `created_at`.
  - Возвращать стандартизированный `{ success, error? }`.

- [x] **3. Интеграция со стабильным пустым поиском:**
  - При пустой выдаче логировать только стабильный запрос длиной не менее 3 символов.
  - Ждать паузу ввода не менее 1.5 секунды перед записью в `Search_Logs`.
  - Не логировать промежуточные live-search запросы по одной-двум буквам.
  - Не логировать restricted-запросы как обычный дефицит.
  - Не менять пользовательский текст пустой выдачи в этом этапе.

- [x] **4. Проверка качества:**
  - Проверить отсутствие незапланированных изменений.
  - Запустить `npm run lint`.
  - Запустить `npm run build`.

## Пользовательская проверка (Пошаговый чеклист)

> Выполните каждый шаг по порядку. Если на любом шаге результат не совпадает с ожидаемым — значит что-то пошло не так.

### Проверка задачи 1: Логирование пустого поиска
1. Запустите приложение командой `npm run dev`.
2. Откройте `http://localhost:3000`.
3. Введите редкий или отсутствующий запрос, например `абракадабра999`, и не меняйте строку минимум 1.5 секунды.
4. ✅ **Ожидаемый результат:** На странице остается обычная заглушка «Препарат не найден», без новых видимых элементов.
5. ❌ **Если не работает:** Проверить `searchProducts(query)` и вызов `logZeroResultSearch`.

### Проверка задачи 2: Запись в базе
1. Откройте Prisma Studio командой `npx prisma studio` или проверьте таблицу `search_logs` через SQL-клиент.
2. Найдите последнюю запись с `search_term = абракадабра999`.
3. ✅ **Ожидаемый результат:** В `search_logs` есть запись с `results_count = 0`, городом Гудермес и временем создания.
4. ❌ **Если не работает:** Проверить миграцию `search_logs`, активный город и Server Action `logZeroResultSearch`.

### Проверка задачи 3: Промежуточные буквы не логируются
1. Введите одну букву, например `т`, дождитесь пустой выдачи.
2. ✅ **Ожидаемый результат:** В `search_logs` не появляется новая запись с `search_term = т`.
3. ❌ **Если не работает:** Проверить минимальную длину запроса перед вызовом `logZeroResultSearch`.

### Проверка задачи 4: Restricted-запрос не попадает в нулевую выдачу
1. Найдите restricted-препарат, например `диазепам`.
2. ✅ **Ожидаемый результат:** Показывается restricted-заглушка, а новая запись дефицита для `диазепам` в `search_logs` не создается.
3. ❌ **Если не работает:** Проверить порядок restricted-проверки и логирования в `searchProducts(query)`.

### Итоговый критерий: Этап завершён, если
- [x] Таблица `search_logs` добавлена миграцией.
- [x] `logZeroResultSearch` записывает нулевую выдачу в базу.
- [x] Пустой поиск на главной логируется без изменения UI только после паузы ввода.
- [x] Запросы короче 3 символов не логируются.
- [x] Restricted-запросы не логируются как дефицит.
- [x] `npm run lint` проходит.
- [x] `npm run build` проходит.
