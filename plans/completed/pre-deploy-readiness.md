# План разработки: Pre-deploy readiness

**Статус:** Завершено (Completed)
**Основание:** [spec/technical/implementation-plan.md](../../spec/technical/implementation-plan.md), [spec/technical/security-performance.md](../../spec/technical/security-performance.md), [spec/features/client-app/client-app-spec.md](../../spec/features/client-app/client-app-spec.md), [spec/features/admin-panel/admin-panel-spec.md](../../spec/features/admin-panel/admin-panel-spec.md), [spec/features/data-architecture/data-architecture.md](../../spec/features/data-architecture/data-architecture.md), [spec/technical/supabase-platform-migration.md](../../spec/technical/supabase-platform-migration.md)

## Цель
Подготовить GotMeds к безопасному пробному деплою и публичному запуску без внедрения CI/CD: закрыть критичные преддеплойные риски, синхронизировать спецификации с фактическим объемом MVP и выполнить воспроизводимые проверки качества.

## Границы этапа
- Входит: SDD-синхронизация спецификаций, перенос CI/CD в будущий отдельный этап, rate limiting публичного поиска, пользовательские жалобы на аптеку и админская модерация жалоб, преддеплойная документация окружения, локальные проверки качества, устранение ложного `404` на SSR-странице продукта при временной ошибке Supabase.
- Не входит: production deploy, `git push`, Pull Request, автоматический CI/CD, новые внешние сервисы, Supabase Storage, Realtime, WhatsApp Business API, платежи, бронирование, личные кабинеты аптек.
- Если по ходу реализации потребуется расширить схему данных или UI сверх пунктов этого плана, план должен быть обновлен и повторно одобрен до продолжения.

## Задачи:
- [x] **1. Синхронизировать спецификации перед деплоем:**
  - Зафиксировать, что CI/CD из этапа деплоя переносится в будущий отдельный план и не является обязательным для текущего пробного деплоя.
  - Разделить незакрытые пункты спецификаций на две группы: обязательные до публичного запуска и будущие улучшения MVP.
  - Явно оставить в будущем объеме: историю поиска, заявку на добавление товара, сортировку "Сначала дешевые", массовый импорт Tier 1 и экспорт отчета дефицита, если они не реализуются в этом этапе.

- [x] **2. Добавить rate limiting публичного поиска:**
  - Реализовать лимит 30 поисковых запросов в минуту с одного IP на серверной границе приложения.
  - Применить лимит к публичному `/api/search`; при необходимости защитить смежный route `/api/pharmacies`, если он может использоваться для массовой выборки.
  - Возвращать понятный ответ `429 Too Many Requests` без раскрытия внутренних деталей Supabase.

- [x] **3. Реализовать жалобы пользователей на аптеку:**
  - Добавить Supabase SQL migration для `pharmacy_reports` и связанных enum-значений согласно спецификациям.
  - Добавить серверное действие отправки жалобы с антиспам-ограничением: не более 1 жалобы на одну аптеку с одного IP за 24 часа.
  - Добавить кнопку "Сообщить об ошибке" в карточку аптеки и варианты жалобы: неверный номер, аптека закрыта, препарата нет в наличии.
  - Добавить защищенный раздел админки для просмотра жалоб, изменения статуса и перехода к редактированию проблемной аптеки.

- [x] **4. Подготовить деплойное окружение и SEO/PWA-проверки:**
  - Добавить безопасный `.env.example` без секретов с обязательными переменными Supabase, сайта и Яндекс.Карт.
  - Проверить, что `sitemap.xml` в production-like окружении получает product URL из Supabase, а при ошибке не ломает сборку.
  - Устранить предупреждение Next.js по изображению продукта, если это не меняет UX и не требует новых внешних сервисов.

- [x] **5. Выполнить преддеплойные проверки качества:**
  - Выполнить `npm run lint`, `npm run build`, `npm run test:unit` и `git diff --check`.
  - При наличии тестовой Supabase базы и админского пользователя выполнить `npm run test:e2e`.
  - После production build/start выполнить `npm run test:lighthouse` и проверить пороги Performance >= 90, SEO >= 95, Accessibility >= 90.

- [x] **6. Устранить ложный `404` на холодном SSR-запросе `/product/[id]`:**
  - Разделить в серверном read-слое состояния "продукт не найден" и "временная ошибка Supabase".
  - Вызывать `notFound()` на `/product/[id]` только для невалидного/отсутствующего продукта согласно `spec/technical/seo-page-spec.md`.
  - При временной ошибке Supabase отдавать не-404 состояние с понятным сообщением без раскрытия внутренних деталей подключения.
  - Обновить или добавить проверку, подтверждающую, что временная ошибка получения продукта не маскируется под `404`.
  - Повторить релевантные проверки качества после исправления.

## Фактический статус проверок
- `npm run lint` — пройден.
- `npm run test:unit` — пройден.
- `git diff --check` — пройден.
- `npm run build` — пройден в production-like окружении с `NEXT_PUBLIC_SITE_URL=http://localhost:3000`; `sitemap.xml` формируется без fallback-ошибки. После перевода шрифта на `next/font` локальная сборка требует сетевой доступ к Google Fonts на этапе build.
- Live-проверка rate limiting — пройдена: 30 запросов получили обычный ответ, следующие запросы вернули `429`.
- `npm run test:e2e` — не запускался: отсутствуют `GOTMEDS_E2E_ADMIN_EMAIL` и `GOTMEDS_E2E_ADMIN_PASSWORD`.
- `npm run test:lighthouse` — пройден после прогрева валидной карточки товара, явного `CHROME_PATH` и точечной оптимизации: Google Font подключается через `next/font`, а данные карточки товара и аналогов кэшируются на сервере на 300 секунд. Финальные scores после задачи 6: `/` — Performance `0.95`, Accessibility `0.95`, SEO `1.00`; `/product/751fd5e0-f031-415d-828d-aac82bef04c3` — Performance `0.95`, Accessibility `1.00`, SEO `1.00`.
- Задача 6 — пройдена: серверное действие `getProductDetails` разделяет `not_found` и `temporary_error`, SSR-страница `/product/[id]` вызывает `notFound()` только для реального отсутствия продукта, а временная ошибка Supabase отображается как не-404 состояние. Добавлены unit-тесты для read-слоя и SSR-ветвления страницы.

## Зафиксированные серые зоны
- Холодный SSR-запрос `/product/[id]` в локальном production-сервере повторно ловил таймаут соединения с Supabase и мог быть отдан как `404`. Серая зона решена в задаче 6 текущего плана: страница продукта отличает реальное отсутствие продукта от временной ошибки Supabase и не маскирует временную ошибку под `404`.

## Пользовательская проверка (Пошаговый чеклист)

> Выполните каждый шаг по порядку. Если на любом шаге результат не совпадает с ожидаемым - значит что-то пошло не так.

### Проверка задачи 1: Спецификации синхронизированы
1. Откройте `spec/technical/implementation-plan.md`.
2. Проверьте раздел деплоя и будущего CI/CD.
3. ✅ **Ожидаемый результат:** CI/CD явно отмечен как будущий отдельный этап, а текущий преддеплойный объем не требует GitHub Actions.
4. ❌ **Если не работает:** остановить реализацию и вернуть план на правку.

### Проверка задачи 2: Rate limiting работает
1. Запустите приложение командой `npm run dev`.
2. Выполните серию запросов к `http://localhost:3000/api/search?q=нурофен` больше лимита за минуту.
3. ✅ **Ожидаемый результат:** первые запросы возвращают обычный JSON-ответ, после превышения лимита API возвращает HTTP `429`.
4. ❌ **Если не работает:** не запускать публичный деплой до исправления лимита.

### Проверка задачи 3: Жалобы отправляются и модерируются
1. Откройте `/map?q=Нурофен`, выберите аптеку и нажмите "Сообщить об ошибке".
2. Выберите тип жалобы и отправьте форму.
3. ✅ **Ожидаемый результат:** пользователь видит сообщение благодарности, повторная жалоба на ту же аптеку в течение суток блокируется.
4. Откройте `/admin/reports` под администратором.
5. ✅ **Ожидаемый результат:** жалоба отображается в списке, ее можно перевести в работу или закрыть, есть переход к редактированию аптеки.
6. ❌ **Если не работает:** проверить Supabase migration, RLS/RPC-права и админскую сессию.

### Проверка задачи 4: SEO/PWA и окружение готовы
1. Проверьте `.env.example`.
2. ✅ **Ожидаемый результат:** файл содержит только имена переменных и безопасные placeholder-значения, без реальных ключей.
3. Запустите `npm run build` с production-like Supabase env.
4. Откройте `/sitemap.xml`.
5. ✅ **Ожидаемый результат:** sitemap содержит главную, карту и URL продуктов из Supabase.
6. ❌ **Если не работает:** проверить `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `NEXT_PUBLIC_SITE_URL` и RPC `gotmeds_get_sitemap_product_ids`.

### Проверка задачи 5: Проверки качества пройдены
1. Выполните `npm run lint`.
2. Выполните `npm run build`.
3. Выполните `npm run test:unit`.
4. При готовой тестовой Supabase базе выполните `npm run test:e2e`.
5. После `npm run build` и `npm run start` выполните `npm run test:lighthouse`.
6. ✅ **Ожидаемый результат:** lint, build и unit-тесты проходят; E2E проходит при настроенном тестовом окружении; Lighthouse соответствует порогам из `lighthouserc.cjs`.
7. ❌ **Если не работает:** этап не считается готовым к публичному запуску.

### Проверка задачи 6: Ложный `404` на SSR-странице продукта устранен
1. Откройте `/product/[id]` для существующего продукта в production-like окружении.
2. ✅ **Ожидаемый результат:** при доступной Supabase база возвращает карточку продукта, а временная ошибка подключения не превращается в страницу `404`.
3. Смоделируйте временную ошибку получения продукта или проверьте соответствующий unit-тест.
4. ✅ **Ожидаемый результат:** временная ошибка Supabase возвращает не-404 состояние с понятным сообщением пользователю.
5. Откройте `/product/[id]` с невалидным или отсутствующим идентификатором.
6. ✅ **Ожидаемый результат:** только в этом случае отображается `404` согласно спецификации SEO-страницы.
7. ❌ **Если не работает:** не завершать этап и проверить ветвление ошибок в read-слое продукта и SSR-странице.
