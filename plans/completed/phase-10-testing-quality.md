# План разработки: Phase 10 Testing & Quality Foundation

**Статус:** Завершён (Completed)
**Основание:** [spec/technical/implementation-plan.md](../../spec/technical/implementation-plan.md), [spec/technical/security-performance.md](../../spec/technical/security-performance.md), [spec/technical/seo-page-spec.md](../../spec/technical/seo-page-spec.md), [spec/technical/supabase-platform-migration.md](../../spec/technical/supabase-platform-migration.md)

## Цель
Закрыть блок «Тестирование & Полировка» Phase 0 MVP: добавить воспроизводимые локальные проверки Jest для Server Actions, Cypress E2E для ключевых пользовательских и админских сценариев, а также Lighthouse-аудит с порогами Performance >= 90, SEO >= 95 и Accessibility >= 90.

## Границы этапа
- Входит: dev-зависимости и конфигурация для Jest, Cypress и Lighthouse; тестовые файлы; fixture для CSV-импорта; npm-скрипты локального запуска; краткая инструкция по обязательным env-переменным для E2E.
- Входит: проверки существующих сценариев поиска, карты, страницы препарата, админского входа, CSV-импорта и SEO/PWA-инфраструктуры.
- Входит: узкий UI-багфикс, найденный при пользовательской проверке админского входа: привести сообщение об ошибке неверного email/пароля к желтому стилю предупреждения, визуально совпадающему с дисклеймером карточки продукта.
- Не входит: прочее изменение пользовательского интерфейса, изменение схемы Supabase, новые SQL-миграции, деплой на production, GitHub Actions CI/CD, Supabase Storage, Realtime или новые внешние интеграции.
- Не входит: хранение реальных секретов, паролей администратора или production-ключей в репозитории.

## Пользовательская проверка (Пошаговый чеклист)

> Выполните каждый шаг по порядку. Если на любом шаге результат не совпадает с ожидаемым — значит что-то пошло не так.

### Проверка задачи 1: Unit-тесты Server Actions
1. Выполните команду `npm run test:unit`.
2. Куда смотреть: терминал.
3. ✅ **Ожидаемый результат:** Jest завершается успешно, а отчет показывает пройденные тесты для поиска, аналогов, аптек и аналитики нулевой выдачи.
4. ❌ **Если не работает:** проверьте текст ошибки, остановите приемку этапа и верните план на исправление конкретного падающего теста.

### Проверка задачи 2: Cypress E2E
1. Убедитесь, что локальная Supabase test-база применена и заполнена `supabase/seed.sql`, а в `.env.local` заданы Supabase URL/ключи.
2. Укажите локальные E2E-переменные для тестового админа способом, который будет описан в итоговой инструкции этапа.
3. В одном терминале запустите приложение командой `npm run dev`.
4. Во втором терминале выполните `npm run test:e2e`.
5. ✅ **Ожидаемый результат:** Cypress проходит сценарии поиска, restricted-поиска, страницы препарата, карты, фильтра «Открыто сейчас», входа в админку и CSV-загрузки.
6. ❌ **Если не работает:** проверьте, что dev-сервер доступен на ожидаемом `baseUrl`, Supabase seed применен, тестовый админ существует в Supabase Auth и доменной таблице `admins`.

### Проверка задачи 3: Lighthouse-аудит
1. Выполните `npm run build`.
2. Запустите production-сервер командой `npm run start`.
3. Во втором терминале выполните `npm run test:lighthouse`.
4. ✅ **Ожидаемый результат:** Lighthouse-аудит для главной и SEO-страницы препарата проходит пороги Performance >= 90, SEO >= 95 и Accessibility >= 90.
5. ❌ **Если не работает:** сохраните отчет Lighthouse, проверьте упавшую категорию и верните этап на доработку до повторной приемки.

### Проверка задачи 4: Единый quality-чек
1. Выполните `npm run test:quality`.
2. Выполните `git diff --check`.
3. ✅ **Ожидаемый результат:** `lint`, `build`, unit-тесты и проверка whitespace проходят без ошибок.
4. ❌ **Если не работает:** этап не считается завершенным; исправьте ошибку и повторите полный quality-чек.

### Проверка задачи 5: Стиль ошибки админского входа
1. Откройте `/admin/login`.
2. Введите неверный email или пароль и отправьте форму.
3. ✅ **Ожидаемый результат:** сообщение `Неверный email или пароль` отображается в желтом предупреждающем блоке с цветами, визуально совпадающими с дисклеймером карточки продукта о противопоказаниях.
4. ❌ **Если не работает:** проверьте CSS-класс ошибки админки и сравните фон, рамку и цвет текста с блоком `.disclaimerBlock` на странице продукта.

### Итоговый критерий: Этап завершён, если

- [x] Cypress E2E закрывает поиск, карту и CSV-загрузку админки.

- [x] `npm run test:quality` проходит без ошибок.
- [x] `git diff --check` проходит без ошибок.
- [x] В репозиторий не попали реальные секреты, пароли, service role key или production-ключи.
