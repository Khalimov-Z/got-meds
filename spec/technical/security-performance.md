## 1. Поисковая оптимизация (SEO)

Так как привлечение трафика планируется без затрат (через органический поиск), SEO — это критический архитектурный элемент.

### 1.1. Генерация Мета-тегов (Next.js Metadata API)
* Для каждой карточки препарата должна быть создана динамическая страница (например, `/product/[id]-nurofen`).
* Сервер Next.js динамически генерирует `title` и `description`.
  * **Title:** "Купить [Название препарата] в Гудермесе — наличие в аптеках, цены"
  * **Description:** "Поиск препарата [Название] в аптеках Гудермеса. Адреса дежурных аптек, цены и наличие на карте города."
* **Open Graph (OG):** Автоматическая генерация OG-изображений для красивого шеринга ссылок в WhatsApp (иконка препарата + текст "Наличие в Гудермесе").

### 1.2. Sitemap и Robots.txt
* Ежедневная автогенерация `sitemap.xml` на основе таблицы `Products`.
* `robots.txt` разрешает индексацию товаров, но запрещает индексацию API-роутов и профилей.

## 2. Progressive Web App (PWA) и Offline

В регионе может быть нестабильный мобильный интернет (Edge/3G). Приложение должно вести себя как нативное.

* **manifest.json:** Должен быть настроен (иконки, цвет темы, `display: "standalone"`), чтобы пользователь мог добавить ярлык `где.таблетка` на экран iPhone/Android. PWA-иконки используют фирменный pin-знак с таблеткой внутри.
* **Service Workers:** 
  * Кэширование статики (CSS, JS, шрифты).
  * Если пользователь теряет сеть, приложение не выдает стандартную ошибку браузера "Нет интернета", а показывает красивый экран оффлайна: *"Похоже, пропала связь. Как только интернет появится, мы продолжим поиск"*.

## 3. Защита и Безопасность

### 3.1. Защита от парсинга конкурентами (Rate Limiting)
Так как наша база — это главный актив, мы должны защитить API поиска от автоматического выкачивания.
* **Решение:** На уровне Nginx или Vercel Edge Middleware устанавливается лимит: не более 30 поисковых запросов в минуту с одного IP.
* **Supabase-миграция:** при переносе поиска в Supabase SDK/RPC rate limiting должен сохраняться на серверной границе приложения, Edge Middleware или другом утвержденном уровне, чтобы публичные RPC не стали каналом массового выкачивания базы.

### 3.2. Анти-спам жалоб
* Жалобы ("Аптека закрыта") записываются в таблицу `Pharmacy_Reports`.
* Чтобы конкуренты не "топили" чужие аптеки, перед записью проверяется `user_ip`. Ограничение: 1 жалоба на конкретную аптеку в 24 часа.

### 3.3. Блокировка запрещенных веществ (Социальный риск)
* Реализована строгая проверка на бэкенде. Если товар имеет флаг `is_social_risk = true`, функция `searchProducts` немедленно прерывается и возвращает пустой массив или спец-флаг `restricted: true`, независимо от того, что написано в БД (никаких обходных путей).

### 3.4. Защита Админ-панели
* Эндпоинты админки (`/admin/*`) защищены Supabase Auth session cookie и серверной проверкой доменной роли администратора. При попытке доступа без действительной сессии пользователя происходит редирект на `/admin/login`.
* Админский вход выполняется через Supabase Auth. Runtime-код приложения не сравнивает пароли напрямую и не хранит пароли в открытом виде.
* Админские операции требуют проверяемую роль администратора (`superadmin`, `content_manager`) и выполняются через защищенные Server Actions/RPC.
* RLS-политики закрывают чувствительные таблицы от `anon` и обычных `authenticated` пользователей.

### 3.5. Supabase RLS и ключи доступа
* Для таблиц, доступных через Supabase API, должен быть включен Row Level Security.
* `anon` ключ может использоваться только в сценариях, где RLS явно разрешает безопасное действие.
* `service_role` ключ запрещено использовать в браузере, клиентских компонентах, публичных bundle-файлах и любых местах, доступных пользователю.
* Операции с `service_role` допускаются только на сервере и только для сценариев, где обход RLS является осознанной частью утвержденной архитектуры.
* SQL functions/RPC с повышенными правами должны проверять входные данные и выполнять только одну ограниченную бизнес-операцию.
* Пользовательский frontend не должен получать прямой доступ к чувствительным таблицам: администраторам, остаткам загрузок, очереди маппинга, жалобам, черному списку и внутренней аналитике.

### 3.6. Конфиденциальность логов поиска и соответствие 152-ФЗ
* **Анонимизация поисковых запросов:** Запросы пользователей в таблице `search_logs` полностью обезличены. Никакие персональные идентификаторы, включая IP-адреса, User-Agent, ID сессий, cookie или ФИО, в логах поиска не сохраняются.
* **Защита геолокации:** Координаты пользователя (`user_latitude`, `user_longitude`) сохраняются исключительно с явного согласия (запрос к геолокации браузера) и автоматически сглаживаются до 3-4 знаков после запятой (точность ~100 метров), что делает невозможным привязку поиска лекарства к конкретной квартире или дому. При отсутствии GPS-координаты сохраняются как `null`.
* **Срок хранения данных:** Логи поиска хранятся в активной базе данных не более **90 дней**. По истечении этого срока записи автоматически удаляются.
* **Доступ к данным:** Статистика дефицитного спроса доступна только авторизованным администраторам с правами модератора/администратора. Доступ к сырой таблице логов из внешней сети полностью закрыт.
* **Связь с ФЗ-152:** Поскольку персональные данные (информация, относящаяся к прямо или косвенно определенному физическому лицу) в процессе логирования поиска не собираются и не хранятся, требования Федерального закона РФ № 152-ФЗ «О персональных данных» на эти логи не распространяются. Логирование носит чисто статистический характер.

## 4. Производительность (Performance)

* **Карта:** Скрипты Яндекс.Карт / 2GIS грузятся асинхронно (через `next/script` со стратегией `lazyOnload` или `worker`), чтобы не блокировать первичную отрисовку интерфейса.
* **Изображения:** Фотографии препаратов пропускаются через компонент `next/image` для автоматического сжатия в формат WebP и "ленивой" загрузки.
