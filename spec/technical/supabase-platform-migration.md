## 1. Назначение документа

Этот документ фиксирует текущее Supabase-состояние GotMeds и правила дальнейшего развития backend/data-платформы.

Код, схема, seed, чтение, админская авторизация, RLS и админские мутации должны развиваться через маленькие независимые планы. Каждый новый этап оформляется отдельным планом в `plans/active/`, проходит явное одобрение пользователя и не расширяет объем работ без обновления плана.

---

## 2. Текущая архитектура

Рабочая архитектура проекта:

```text
Next.js App Router
  -> Server Components / Server Actions / Route Handlers
    -> Supabase SDK / SQL functions / RPC
      -> Supabase Auth
      -> Supabase PostgreSQL
      -> RLS policies
```

Текущие роли компонентов:

- `Next.js` отвечает за клиентское приложение, SEO-страницы, карту, админку и серверную композицию сценариев.
- `Supabase PostgreSQL` является основной базой данных проекта.
- `Supabase SQL migrations` в `supabase/migrations/` являются источником изменений схемы, RLS и RPC.
- `supabase/seed.sql` является тестовым seed-слоем для чистой Supabase PostgreSQL базы.
- `Supabase Auth` является механизмом админской авторизации.
- `RLS policies` закрывают чувствительные таблицы от публичного доступа.
- `SQL functions / RPC` используются для сложной серверной логики рядом с данными: fuzzy search, выборка аптек, аналоги, агрегации аналитики и атомарные админские операции.
- `Supabase Storage` используется только после отдельного утвержденного плана, если проекту потребуется хранение медиа.

---

## 3. Строгие правила Supabase-слоя

1. `service_role` ключ Supabase запрещено использовать в браузере, клиентских компонентах и публичном JavaScript.
2. Пользовательский frontend не должен получать прямой доступ к чувствительным таблицам.
3. Админские операции должны требовать проверенную роль администратора (`superadmin`, `content_manager`).
4. SQL functions/RPC с повышенными правами должны иметь минимальный объем ответственности и валидацию входных данных.
5. Существующие пользовательские сценарии должны сохранять поведение: поиск, карта, страница продукта, аналоги, админка, CSV-импорт, маппинг, черный список и аналитика.
6. Любое изменение схемы данных выполняется только через утвержденный план и должно быть совместимо с текущими production-данными.
7. Подключение Supabase Storage, Realtime или новых внешних интеграций требует отдельной спецификации или отдельного утвержденного плана.

## 4. Что покрывает Supabase

| Область | Текущее состояние |
| --- | --- |
| База данных | Supabase PostgreSQL |
| Миграции схемы | Supabase SQL migrations |
| Seed тестовых данных | `supabase/seed.sql` |
| Доступ к данным | Supabase SDK, SQL functions и RPC |
| Fuzzy search | `pg_trgm` в Supabase PostgreSQL через SQL/RPC |
| Админская авторизация | Supabase Auth с админскими ролями |
| Защита данных | RLS policies плюс серверные проверки |
| Файлы и медиа | Supabase Storage только после отдельного плана |
| Realtime | Не используется |

## 5. Завершенные этапы перехода

### 5.1. Supabase PostgreSQL foundation

**Результат:** GotMeds подключен к Supabase PostgreSQL как основной managed PostgreSQL.

**Проверено:** расширение `pg_trgm`, индексы, базовые сценарии приложения и сборка.

### 5.2. Supabase SQL migrations and seed

**Результат:** схема и тестовые данные воспроизводятся через `supabase/migrations/` и `supabase/seed.sql`.

**Проверено:** создание чистой тестовой Supabase PostgreSQL базы с нуля.

### 5.3. Supabase read layer

**Результат:** read-сценарии поиска, страницы продукта, аналогов, карты, sitemap и дашборда дефицита выполняются через Supabase SDK/RPC.

**Проверено:** пользовательские контракты UI не изменились.

### 5.4. Supabase Auth and RLS

**Результат:** админская авторизация работает через Supabase Auth, а RLS закрывает чувствительные операции.

**Проверено:** обычный пользователь не имеет доступа к админским операциям.

### 5.5. Supabase admin mutations

**Результат:** CRUD аптек, CSV-импорт остатков, маппинг alias, игнорирование строк, черный список и запись аналитики выполняются через Supabase SDK/RPC.

**Проверено:** админские операции проходят ручную проверку без изменения UI.

### 5.6. Legacy database decommission

**Результат:** legacy database-файлы, прямые DB-клиенты, устаревшие зависимости и вводящие в заблуждение комментарии удаляются отдельным утвержденным планом.

**Критерий завершения:** проект собирается и проходит проверки без legacy database-слоя.

---

## 6. Требования к проверке

Перед завершением любого следующего Supabase-этапа должны быть проверены:

- поиск с опечатками и алиасами;
- restricted-поиск для товаров социального риска;
- страница продукта и SEO metadata;
- карта и список аптек;
- фильтр "Открыто сейчас";
- админский вход;
- CSV-импорт и маппинг;
- черный список;
- дашборд дефицитных позиций;
- `npm run lint`;
- `npm run build`;
- `git diff --check`.

---

## 7. Что не входит без отдельного плана

Следующие изменения не внедряются без отдельной спецификации или отдельного утвержденного плана:

- изменение структуры таблиц;
- изменение пользовательского интерфейса;
- подключение Supabase Storage;
- внедрение Realtime;
- новые внешние интеграции;
- деплой на production.
