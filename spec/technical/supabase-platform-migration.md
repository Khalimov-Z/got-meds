## 1. Назначение документа

Этот документ фиксирует стратегию безопасного перехода GotMeds на Supabase как целевую backend/data-платформу.

Миграция должна выполняться маленькими независимыми этапами. Каждый этап оформляется отдельным планом в `plans/active/`, проходит явное одобрение пользователя и не расширяет объем работ без обновления плана.

---

## 2. Текущая архитектура

На момент принятия этой стратегии рабочая архитектура проекта:

```text
Next.js App Router
  -> Server Components / Server Actions / Route Handlers
    -> Prisma Client
      -> PostgreSQL
```

Текущие роли компонентов:

- `Next.js` отвечает за клиентское приложение, SEO-страницы, карту, админку и серверные действия.
- `Prisma` отвечает за типизированный доступ к БД, схему `prisma/schema.prisma`, миграции и seed.
- `PostgreSQL` хранит доменные данные GotMeds и выполняет `pg_trgm`-поиск.
- Админская авторизация реализуется текущим серверным механизмом проекта, без Supabase Auth.

Эта архитектура остается допустимым переходным состоянием до завершения всех утвержденных этапов миграции.

---

## 3. Целевая архитектура

Целевая архитектура после полной миграции:

```text
Next.js App Router
  -> Supabase SDK / Server Actions / RPC
    -> Supabase Auth
    -> Supabase PostgreSQL
    -> RLS policies
    -> SQL functions / RPC
    -> Supabase Storage
```

Целевые роли компонентов:

- `Next.js` остается приложением для UI, SEO, карты, админки и серверной композиции сценариев.
- `Supabase PostgreSQL` становится основной базой данных проекта.
- `Supabase SQL migrations` становятся источником изменений схемы после отдельного этапа переноса миграций.
- `Supabase Auth` становится целевым механизмом админской авторизации после отдельного этапа миграции auth.
- `RLS policies` становятся обязательным слоем защиты таблиц, доступных через Supabase API.
- `SQL functions / RPC` используются для сложной серверной логики рядом с данными: fuzzy search, выборка аптек, агрегации аналитики, атомарные админские операции.
- `Supabase Storage` используется для будущего хранения медиа, если это будет утверждено отдельной спецификацией или планом.

---

## 4. Строгие правила миграции

1. Prisma удаляется только на финальном этапе, когда все runtime-запросы, миграции и seed имеют проверенный Supabase-эквивалент.
2. Нельзя одновременно переписывать весь data layer, авторизацию и админские мутации одним планом.
3. Каждый этап должен иметь rollback-подход, позволяющий вернуться к предыдущему рабочему состоянию.
4. `service_role` ключ Supabase запрещено использовать в браузере, клиентских компонентах и публичном JavaScript.
5. До готовности RLS-политик пользовательский frontend не должен получать прямой доступ к чувствительным таблицам.
6. Админские операции остаются за серверной границей до тех пор, пока не настроены Supabase Auth, роли администратора и RLS-политики.
7. Существующие пользовательские сценарии должны сохранять поведение: поиск, карта, страница продукта, аналоги, админка, CSV-импорт, маппинг, черный список и аналитика.
8. Любое изменение схемы данных выполняется только через утвержденный план и должно быть совместимо с текущими production-данными.

---

## 5. Что покрывает Supabase

| Область | Текущее состояние | Целевое состояние |
| --- | --- | --- |
| База данных | PostgreSQL через Prisma | Supabase PostgreSQL |
| Миграции схемы | Prisma migrations | Supabase SQL migrations |
| Доступ к данным | Prisma Client | Supabase SDK, SQL functions и RPC |
| Fuzzy search | `pg_trgm` в PostgreSQL | `pg_trgm` в Supabase PostgreSQL через SQL/RPC |
| Админская авторизация | текущий серверный механизм проекта | Supabase Auth с админскими ролями |
| Защита данных | серверные проверки приложения | RLS policies плюс серверные проверки |
| Файлы и медиа | внешнее S3-подобное хранилище в будущем | Supabase Storage после отдельного плана |
| Realtime | не используется | не внедряется без отдельной необходимости |

## 6. Этапы миграции

### 6.1. Supabase PostgreSQL foundation

**Цель:** использовать Supabase как управляемый PostgreSQL, не меняя Prisma и runtime-код приложения.

**Границы работ:**

- Создать Supabase-проект.
- Проверить connection string для миграций и runtime-подключения.
- Проверить поддержку `pg_trgm`.
- Прогнать текущие Prisma migrations на Supabase PostgreSQL.
- Проверить seed и базовые пользовательские сценарии.

**Критерий перехода дальше:** приложение работает через Supabase PostgreSQL без изменения data layer.

**Rollback:** вернуть `DATABASE_URL` на предыдущую PostgreSQL-базу.

### 6.2. Supabase SQL migrations and seed

**Цель:** перенести источник схемы и начальных данных из Prisma в Supabase SQL-подход.

**Границы работ:**

- Перевести существующую Prisma-схему в SQL-миграции.
- Перенести enum, foreign keys, индексы, `pg_trgm` extension и ограничения.
- Подготовить Supabase seed для тестовых данных.
- Проверить создание чистой базы с нуля.

**Критерий перехода дальше:** новая Supabase-база воспроизводится SQL-миграциями и seed без потери структуры.

**Rollback:** оставить Prisma migrations источником схемы до исправления SQL-миграций.

### 6.3. Supabase read layer

**Цель:** перенести чтение данных на Supabase без изменения пользовательского поведения.

**Границы работ:**

- Перенести чтение страницы продукта.
- Перенести поиск препаратов.
- Перенести аналоги.
- Перенести выборку аптек для карты.
- Перенести чтение аналитики дефицита.
- Для сложных запросов использовать SQL functions/RPC.

**Критерий перехода дальше:** все read-сценарии проходят проверку и не используют Prisma в runtime.

**Rollback:** вернуть конкретный read-сценарий на Prisma-реализацию.

### 6.4. Supabase Auth and RLS

**Цель:** перенести админскую авторизацию и правила доступа на Supabase Auth и RLS.

**Границы работ:**

- Спроектировать модель админских ролей: `superadmin`, `content_manager`.
- Связать Supabase Auth users с доменным профилем администратора.
- Включить RLS для таблиц, доступных через Supabase API.
- Настроить политики чтения публичных данных.
- Настроить политики админских операций.
- Проверить, что обычный пользователь не имеет доступа к админским операциям.

**Критерий перехода дальше:** админка работает через Supabase Auth, а RLS закрывает чувствительные операции.

**Rollback:** оставить текущую серверную авторизацию до устранения проблем с Supabase Auth/RLS.

### 6.5. Supabase admin mutations

**Цель:** перенести админские операции на Supabase SDK/RPC.

**Границы работ:**

- Перенести CRUD аптек.
- Перенести CSV-импорт остатков.
- Перенести маппинг alias.
- Перенести черный список.
- Перенести модерацию жалоб после реализации самой функции жалоб.
- Перенести запись аналитики и админские агрегации, если они еще используют Prisma.

**Критерий перехода дальше:** админские операции выполняются через Supabase-слой и проходят ручную проверку.

**Rollback:** вернуть отдельную mutation на Prisma-реализацию.

### 6.6. Prisma decommission

**Цель:** удалить Prisma как зависимость и источник схемы только после полного функционального паритета.

**Границы работ:**

- Убедиться, что в runtime-коде нет импортов Prisma.
- Убедиться, что схема и seed воспроизводятся Supabase-инструментами.
- Удалить Prisma-зависимости и Prisma-клиент.
- Удалить или архивировать `prisma/` согласно отдельному плану.
- Обновить scripts и документацию запуска.

**Критерий завершения:** проект собирается и проходит проверки без Prisma.

**Rollback:** не удалять Prisma до завершения всех предыдущих этапов.

## 7. Требования к безопасности

- Все таблицы, доступные через Supabase API, должны иметь включенный RLS.
- Публичные политики должны разрешать только те чтения, которые нужны клиентскому приложению.
- Запись жалоб, логов поиска и других пользовательских событий должна иметь антиспам-ограничения.
- Админские операции должны требовать проверенную роль администратора.
- `service_role` используется только на сервере и только для операций, где RLS намеренно обходится контролируемым образом.
- SQL functions с повышенными правами должны иметь минимальный объем ответственности и валидацию входных данных.

---

## 8. Требования к проверке

Перед переходом к следующему этапу миграции должны быть проверены:

- поиск с опечатками и алиасами;
- restricted-поиск для товаров социального риска;
- страница продукта и SEO metadata;
- карта и список аптек;
- фильтр "Открыто сейчас";
- админский вход;
- CSV-импорт и маппинг;
- черный список;
- дашборд дефицитных позиций;
- `npm run lint`;
- `npm run build`.

---

## 9. Что не входит в эту стратегию

Эта стратегия не утверждает немедленную реализацию следующих изменений:

- удаление Prisma;
- переход на Supabase Auth;
- включение RLS;
- переписывание Server Actions;
- изменение структуры таблиц;
- изменение пользовательского интерфейса;
- подключение Supabase Storage;
- внедрение Realtime;
- деплой на production.

Каждый из этих пунктов требует отдельного плана и явного одобрения пользователя.
