## 1. Обзор Архитектуры

Для Фазы 0 (MVP) выбрана **монорепозиторная архитектура (Monorepo)**. Это означает, что клиентское приложение (Frontend), админ-панель и серверная логика (Backend API) будут находиться в единой кодовой базе. Это обеспечит максимальную скорость разработки ИИ-агентом, единую типизацию и простоту развертывания.

**Основной язык программирования:** `TypeScript` (Строгая типизация защитит нас от множества багов при связывании БД и фронтенда).

---

## 2. Frontend (Пользовательский интерфейс)

* **Фреймворк:** `Next.js` (App Router).
  * *Обоснование:* Next.js критически важен для SEO (рендеринг на стороне сервера - SSR). Это позволит поисковикам (Яндекс/Google) индексировать страницы конкретных лекарств ("Купить Нурофен Гудермес"), что обеспечит нам бесплатный органический трафик.
* **Библиотека интерфейсов:** `React.js` (встроен в Next.js).
* **Стилизация (Дизайн):** `Vanilla CSS` (CSS Modules).
  * *Обоснование:* В соответствии с нашими архитектурными стандартами, для достижения максимально премиального, "вау" эффекта и полного контроля над микро-анимациями (glassmorphism, hover-эффекты) используется чистый CSS. Мы не используем TailwindCSS, чтобы интерфейс не выглядел шаблонно.
* **Интеграция Карт:** `Яндекс.Карты API` (Yandex Maps JS API) или `2GIS API`. Обеспечивают наилучшую детализацию для Кавказа и малых аптек.
* **Иконки:** `Lucide React` или кастомные SVG для премиального вида.

---

## 3. Backend (Серверная логика и API)

* **Среда выполнения:** `Node.js` (в рамках Next.js API Routes / Server Actions).
  * *Обоснование:* Позволяет ИИ писать бэкенд на том же языке (TypeScript), что и фронтенд, переиспользуя типы базы данных.
* **Аутентификация (Для Админки):** на текущем этапе используется серверная авторизация проекта (`NextAuth.js` или кастомная JWT-реализация) для защиты роутов админ-панели `/admin/*`.
  * *Целевое состояние:* после отдельного утвержденного этапа миграции админская авторизация должна перейти на `Supabase Auth` с админскими ролями и RLS-политиками, согласно [стратегии миграции на Supabase](./supabase-platform-migration.md).
* **Обработка CSV:** Библиотека `csv-parser` или `papaparse` для стримингового парсинга больших прайс-листов в админке без перегрузки памяти сервера.

## 4. База Данных и Данные

* **База данных:** `PostgreSQL`.
  * *Обоснование:* Самая надежная реляционная БД.
  * *Целевое состояние:* основной PostgreSQL должен быть размещен в Supabase, чтобы проект постепенно перешел на единую backend/data-платформу.
* **Поисковый движок (Fuzzy Search):** Расширение `pg_trgm` внутри PostgreSQL.
  * *Обоснование:* Позволяет реализовать защиту от опечаток (триграммный поиск) без необходимости разворачивать отдельные сложные решения вроде ElasticSearch или Meilisearch на Фазе 0, что снижает стоимость серверов до нуля.
* **ORM / Data Access:** `Prisma` является текущим рабочим и переходным слоем доступа к БД.
  * *Обоснование текущего состояния:* Prisma обеспечивает типизацию для TypeScript и поддерживает уже реализованные запросы, миграции и seed.
  * *Целевое состояние:* Prisma должен быть заменен на Supabase SDK, SQL functions/RPC и Supabase SQL migrations только по этапам, описанным в [supabase-platform-migration.md](./supabase-platform-migration.md). Удаление Prisma допустимо только финальным отдельным планом после проверки функционального паритета.

---

## 5. Инфраструктура и Развертывание (Hosting)

Поскольку бюджет Фазы 0 минимален (в идеале нулевой), мы используем Serverless-инфраструктуру:

* **Хостинг Приложения:** `Vercel` (Бесплатный тариф для MVP). Идеально интегрирован с Next.js, обеспечивает автоматический CI/CD (деплой при коммите в GitHub) и быструю доставку контента (CDN).
* **Хостинг Базы Данных:** `Supabase` как целевая managed PostgreSQL-платформа проекта.
* **Backend/Data-платформа:** `Supabase` постепенно берет на себя PostgreSQL, Auth, RLS, SQL functions/RPC и Storage в рамках отдельных утвержденных планов.
* **Хранение медиа:** Если в будущем потребуются фото препаратов, целевым вариантом является `Supabase Storage`, но подключение Storage выполняется только отдельным планом.

## 6. Структура Репозитория (Примерная)

```text
/
├── src/
│   ├── app/                # Роуты Next.js (страницы сайта)
│   │   ├── (client)/       # Клиентская часть (поиск, карта)
│   │   └── (admin)/        # Админ-панель
│   ├── components/         # Переиспользуемые UI-компоненты (Кнопки, Карточки)
│   ├── lib/                # Утилиты, текущий Prisma-слой и будущий Supabase data layer
│   ├── styles/             # Глобальные стили (Vanilla CSS) и CSS-модули
├── prisma/
│   └── schema.prisma       # Текущая переходная схема БД до переноса на Supabase SQL migrations
└── package.json
```
