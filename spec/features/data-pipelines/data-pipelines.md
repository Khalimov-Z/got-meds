## 1. Формирование Эталонной Базы
**Источник:** Apteka.ru (или аналог)
* **Процесс:** Одноразовый (или редкий, 1 раз в месяц) парсинг каталога.
* **Цель:** Собрать чистые названия, фото, статус (рецепт/не рецепт) и действующие вещества для умных аналогов.
* **Категоризация:** Скрипт должен классифицировать товары по полю `category` (Лекарства, Медтехника, Витамины, Мать и дитя). Эта база является стержнем, вокруг которого строится вся остальная логика платформы.

## 2. Интеграция Сетевых Аптек (Парсинг, напр. "Апрель", Future)
**Источник:** API или парсинг сайта сети
* **Тип запуска:** Автоматический Cron Job (например, каждую ночь в 03:00).
* **Алгоритм (Full Sync):** 
  1. **Сброс (Out of Stock):** Скрипт *сбрасывает* (удаляет) все текущие остатки данной аптеки в нашей БД перед загрузкой новых. Это критично, чтобы проданные за день товары исчезли с карты.
  2. **Маппинг:** Скрипт сопоставляет товар из сети с нашей эталонной базой (желательно по штрихкоду EAN или точному текстовому совпадению).
  3. **Запись:** Для найденных совпадений обновляет таблицу `Inventory`, устанавливая статус `in_stock`.
  4. Ставит `updated_at` = текущее время.

> Статус pre-deploy readiness: автоматический парсинг сетевых аптек является будущей внешней интеграцией и не входит в текущий этап. Для пробного деплоя данные воспроизводятся через Supabase SQL migrations/seed и админские CSV-сценарии.

## 3. Интеграция Частных Аптек (Tier 2)
**Источник:** Ручная выгрузка из 1С / МойСклад в формате Excel/CSV.
* **Тип запуска:** Асинхронно, по мере получения файла от владельца аптеки (загружается через Админ-панель).
* **Алгоритм (Full Sync & Mapping):**
  1. **Сброс (Out of Stock):**  Аналогично сетям, удаляем все старые остатки этой аптеки.
  2. Чтение CSV (колонки: "Название", "Остаток").
  3. **Проверка алиасов:** Поиск совпадений по таблице `Product_Aliases` (словарь синонимов 1С).
  4. **Обновление БД:** Для найденных совпадений пишем в `Inventory` со статусом `likely_in_stock` (Вероятно в наличии).
  5. **Очередь нераспознанных товаров:** Все строки из CSV, для которых не нашлось алиаса, помещаются во временную таблицу (очередь маппинга). Модератор в Админке позже вручную привяжет эти кривые строки к эталонным товарам, и алгоритм при следующем запуске уже узнает их автоматически.

## 4. Малые Аптеки (Tier 1 - Киоски)
**Источник:** Сбор базы вручную (обход города, прозвон) + открытые источники.
* **Тип запуска:** Вручную через интерфейс добавления аптеки в Админ-панели GotMeds.
* **Алгоритм:** Для них принципиально не загружаются остатки. Система генерирует статус `unknown` динамически (в момент поиска пользователя) только для базовых препаратов (прошедших Смарт-фильтр, например цена < 1000 руб и без рецепта).

## 5. Обработка сбоев и отказоустойчивость (Fault Tolerance)
Для предотвращения ситуаций, когда сбой парсинга приводит к ложному обнулению остатков аптек на сайте («вчера товары были, сегодня аптека пустая»), внедряется следующая логика защиты:

1. **Обнаружение ошибок парсинга (Критерии сбоя):**
   * **Технические сбои:** Неверный формат файла, отсутствие обязательных колонок ("Название", "Остаток"), ошибки чтения CSV-парсером или падение транзакции базы данных.
   * **Логические сбои (Смена формата/Аномалии):** Структура файла корректна, но количество распознанных товаров (`recognizedRows`) равно `0` или аномально снизилось (более чем на 50%) по сравнению с текущими остатками этой аптеки в базе данных.
2. **Безопасный порог (Threshold):**
   * Если текущее количество остатков аптеки в БД составляет более 10 товаров, а при новом импорте распознано `0` позиций или падение объема товаров превышает 50%, система считает импорт потенциально ошибочным.
   * Процесс импорта и вызов RPC-функции `gotmeds_admin_full_sync_inventory` блокируются.
   * Для обхода этой блокировки администратор должен явно установить флаг «Принудительная загрузка» (чекбокс `forceUpload` на странице загрузки).
3. **Сохранение данных предыдущего импорта:**
   * При возникновении технического сбоя или блокировки по threshold транзакция импорта прерывается. Старые данные в таблице `inventory` полностью сохраняются.
4. **Оповещения (Alerting):**
   * Модератор, выполняющий загрузку, видит детальное сообщение об ошибке на странице импорта (сколько товаров распознано, какой порог нарушен, предложение использовать принудительную загрузку).
   * Ошибки импорта фиксируются в серверных логах приложения.
5. **Поведение для пользователей (Frontend):**
   * Пользователи продолжают искать и видеть товары в наличии на карте и страницах по старым (вчерашним) данным. 
   * На карточке аптеки и карте отображается отметка времени последнего успешного обновления остатков (`inventory.updated_at`), чтобы пользователь понимал степень актуальности информации.
