Данный документ описывает логику работы поискового движка, правила фильтрации, сортировки и выдачи результатов.

## 1. Базовый поиск и обработка запросов

### 1.1 Fuzzy Search (Защита от опечаток)
* **Проблема:** Пользователи часто пишут названия лекарств с ошибками, особенно в состоянии стресса или на ходу.
* **Решение:** Поиск должен прощать опечатки (Fuzzy Search). Ввод "аспирин", "асперен" или "осперен" должен выдавать эталонный "Аспирин".
* **Техническая реализация:** Использование триграмм (например, расширение `pg_trgm` в PostgreSQL) или выделенный движок (Meilisearch / Typesense).

### 1.2 Синонимы и Торговые наименования (Серая зона: решено)
* **Сценарий:** Пользователь ищет "Дротаверин" (действующее вещество). 
* **Логика:** Система должна выдавать как сам "Дротаверин", так и его популярные торговые наименования (например, "Но-шпа"), опираясь на связь через `active_ingredient`.

### 1.3 Парсинг дозировок и форм (Серая зона: решено)
* **Сценарий:** Запрос содержит форму выпуска: "Аспирин 500мг таблетки".
* **Логика:** Поисковик должен приоритизировать результаты, в которых поля `dosage` (500мг) и `form` (таблетки) совпадают с запросом, а не просто искать точное совпадение по всему названию.

## 2. Логика выдачи Аналогов

Когда пользователь нажимает кнопку "Аналоги" у препарата `X`, система ищет альтернативы по следующему алгоритму:
1. Извлекается `active_ingredient` (действующее вещество) препарата `X`.
2. Система находит препараты с таким же `active_ingredient`, такой же `category` и такой же `form`.
3. Совпадение `dosage` повышает релевантность, но не является обязательным фильтром.

Аналоги не должны смешивать разные пользовательские назначения. Детские формы и товары категории `mother_and_baby` не считаются аналогами взрослых препаратов категории `medicine`, даже если действующее вещество совпадает.

### 2.1 Умная фильтрация аналогов (Устранение противоречия)
* **Проблема:** Если фильтровать аналоги *только* по наличию в таблице `Inventory`, мы отсечем малые аптеки (Tier 1), у которых нет учета остатков, даже для базовых препаратов.
* **Решение:**
  * **Для списка аналогов:** аналог предлагается пользователю только если он есть в таблице `Inventory` у активной аптеки Tier 2 или Tier 3 со статусом `IN_STOCK` или `LIKELY_IN_STOCK`.
  * **Для карты выбранного аналога:** после перехода пользователя на карту применяется общий смарт-фильтр малых аптек из раздела 3.

## 3. Смарт-фильтр Малых Аптек (Tier 1)

Это ключевой механизм защиты лояльности пользователей. Мы не должны отправлять людей в ларьки за редкими препаратами.

* Когда пользователь ищет препарат, система проверяет его свойства:
  * `is_prescription == true` (По рецепту) ИЛИ
  * `price_estimate > 1000` (Дорогой препарат) ИЛИ
  * `price_estimate == null` (Цена неизвестна — перестраховываемся).
* **Если Да (Редкий/Сложный препарат):** На карте показываются **ТОЛЬКО** сетевые аптеки и крупные частные (Tier 2/3). Малые аптеки (Tier 1) полностью скрываются.
* **Если Нет (Базовый препарат массового спроса):** На карте отображаются **ВСЕ** аптеки, включая малые аптеки-киоски (Tier 1). Для Tier 1 выводится статус "❓ Наличие неизвестно" и кнопка "Спросить в WhatsApp".
  * *Логика:* Шанс найти парацетамол в ларьке у дома высок, шанс найти специфический инсулин — равен нулю.

## 4. Фильтрация по времени работы и Гео-сортировка

### 4.1 Статус "Открыто сейчас" (Динамический фильтр)
* Вместо жесткого фильтра по времени (например, с 22:00 до 08:00) система динамически проверяет статус каждой аптеки на основе ее расписания (`working_hours` — JSONB с графиком по дням недели).
* При поиске по умолчанию применяется тогл **"Открыто сейчас"**: на карте отображаются только работающие в данный момент аптеки (включая круглосуточные `is_24_7 = true`).
* **Учет региональных особенностей (Серая зона: решено):** Расписание должно поддерживать дневные перерывы (например, на обед или на пятничную молитву — Рузба в Чечне). 
* **Отключаемость фильтра:** Пользователь должен иметь возможность **отключить** фильтр "Открыто сейчас" (например, если он ищет лекарство ночью, чтобы пойти за ним утром). Закрытые аптеки в таком случае отображаются серым цветом (затененными).

### 4.2 Сортировка выдачи
* **Основная сортировка:** В списке (Bottom Sheet) аптеки всегда сортируются по **расстоянию** от пользователя (от ближайшей к дальней), если дано разрешение на GPS.
* **Без GPS (Серая зона: решено):** Если пользователь запретил доступ к геоданным, аптеки сортируются по принадлежности к Tier (сначала Tier 3 сетевые, затем Tier 2, затем Tier 1) или по алфавиту в рамках города.
* **Сортировка по цене:** Если для препарата есть данные о ценах из `Inventory` (Tier 2/3 аптеки), пользователю доступен переключатель сортировки "Сначала дешевые".

## 5. Обработка "Нулевой выдачи" (Empty State)

Система различает две ситуации, когда пользователь не получает результатов:

### 5.1 Препарат не найден в Справочнике (Not in DB)
* **Проблема:** Пользователь ввел название, и система ничего не нашла даже с Fuzzy Search (препарата нет в эталонной базе).
* **Решение:**
  1. Вывести сообщение: *"Препарат не найден. Возможно, он снят с производства или мы еще не добавили его в базу."*
  2. Предложить кнопку **"Оставить заявку на добавление"** (запрос отправляется в админку/Telegram бот администратора).
  3. **Аналитика:** Запрос логируется в базу данных для пополнения каталога контент-менеджерами.

### 5.2 Тотальный дефицит (Out of stock)
* **Проблема:** Препарат найден в Справочнике, но его нет в наличии ни в одной аптеке города.
* **Решение:**
  1. Вывести сообщение: *"Сейчас нет в наличии в Гудермесе."*
  2. Предложить кнопки WhatsApp дежурных аптек для запроса препарата "под заказ".
  3. Предложить аналоги (если применимо).
  4. **Аналитика:** Запрос логируется как дефицит для формирования отчетов B2B (неудовлетворенный спрос).
