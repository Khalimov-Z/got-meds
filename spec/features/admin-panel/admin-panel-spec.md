## 1. Обзор
Интерфейс для Администратора платформы. Служит для поддержания актуальности данных, обработки прайсов частных аптек (Tier 2), управления сетью киосков (Tier 1), модерации контента и сбора аналитики для B2B-продаж.

Точка входа в админ-панель доступна по адресу `/admin`. На клиентской главной странице `/` допускается компактная служебная ссылка "Администратор" в верхней зоне рядом с брендом. Ссылка не размещается в футере и не содержит форму логина; при отсутствии сессии защита `/admin/*` перенаправляет администратора на `/admin/login`. Административные экраны используют тот же публичный бренд-блок `где.таблетка` с переходом на `/`, не переименовывая внутренний проект GotMeds.

В рамках перехода на Supabase текущая серверная авторизация админки остается допустимым временным состоянием. Целевое состояние — Supabase Auth с ролями `superadmin` и `content_manager`, RLS-политиками и серверной проверкой админских операций. Этот переход выполняется только отдельным утвержденным планом и не должен смешиваться с переносом всех админских мутаций.

## 2. Управление Справочником Аптек

### 2.1 CRUD операции (Создание, Чтение, Обновление, Удаление)
* **Добавление/Редактирование:** Админ может добавлять аптеки, указывая: Название, Гео-координаты, Тип (Tier 1/2/3), График работы (включая перерывы), номер WhatsApp.
* **Управление статусом (Серая зона: решено):** Аптеки не удаляются из базы физически (чтобы не ломать аналитику). Используется статус (Active / Paused / Closed). Если аптека закрылась на ремонт, админ переводит её в `Paused`, и она пропадает с карты.
* **Массовый импорт Tier 1 (Future):** Возможность загрузить CSV-список аптек с координатами (спарсенный из 2GIS/Яндекс.Карт), чтобы быстро заполнить карту малыми аптеками без остатков. Не входит в текущий pre-deploy readiness, так как справочник аптек уже поддерживает ручное создание и редактирование.

## 3. Загрузка и Маппинг Прайс-листов (Tier 2)

Это критический функционал. Прайсы из 1С частных аптек часто содержат сокращения, ошибки и нестандартные названия.

### 3.1 Алгоритм загрузки (Серая зона: решено)
1. Админ загружает CSV/Excel файл прайса для конкретной аптеки.
2. **Полное замещение:** Новый прайс полностью перезаписывает (или помечает старыми) предыдущие остатки данной аптеки в таблице `Inventory`, чтобы исключить "фантомные" препараты, которые уже распроданы.

### 3.2 Процесс маппинга (Сопоставления)
1. Система автоматически ищет 100% совпадения с Эталонной базой препаратов (например, спарсенной с Apteka.ru).
2. Система применяет словарь синонимов (Aliases), созданный ранее.
3. Для несовпавших позиций выводится таблица "Требует внимания".

### 3.3 Ручной маппинг и обработка ошибок (Управление синонимами)
* **Связывание:** Админ через поиск находит в Эталонной базе нужный препарат и нажимает "Связать". Система запоминает этот Alias для будущих загрузок.
* **Игнорирование:** Если в прайсе оказался мусор (например, "Пакет-майка"), админ нажимает "Игнорировать". Этот Alias тоже запоминается, чтобы в будущем этот товар просто отбрасывался.
* **Управление Alias-ами (Исправление ошибок):**
  * **Кто может отменить/изменить маппинг:** Любой авторизованный администратор или модератор с ролью в системе (доступ контролируется через RLS-политики базы данных на таблицу `product_aliases` и серверную проверку сессии на бэкенде). Обычные клиенты не имеют доступа к изменению этих связей.
  * **Интерфейс:** Администратору доступен реестр созданных синонимов (алиасов) на странице `/admin/aliases` с возможностью поиска по сырой строке, названию препарата и удаления ошибочно созданной связи.
  * **Регламент исправления ошибок сопоставления:**
    1. Если сырая строка CSV была ошибочно связана модератором не с тем препаратом, это приводит к некорректным остаткам "в наличии" в таблице `inventory`.
    2. Администратор переходит в `/admin/aliases` и удаляет ошибочную связь (запись удаляется из `product_aliases`).
    3. **Влияние на остатки:** Удаление синонима не удаляет остатки мгновенно (поскольку в таблице `inventory` хранятся только ID препарата и ID аптеки, а не исходная сырая строка).
    4. Чтобы убрать ошибочное наличие препарата с сайта мгновенно, в интерфейсе удаления выводится предупреждение с перечнем аптек, загружавших эту позицию, чтобы администратор мог зайти в раздел загрузки остатков и повторно импортировать прайс-лист для затронутой аптеки. При повторной загрузке ошибочно сопоставленная строка больше не распознается по синониму, попадает в очередь неразмеченных на страницу `/admin/mapping`, а некорректный остаток удаляется из `inventory`.
  * **История изменений:** Вся история ручного сопоставления и удаления синонимов логируется на сервере в системных логах GotMeds с фиксацией ID администратора, выполнившего действие, времени и параметров измененной связи.

## 4. Модерация и Безопасность

### 4.1 Обработка жалоб пользователей
* В админку падают репорты из клиентского приложения (например, "Аптека закрыта", "Препарата нет в наличии", "Неправильный WhatsApp").
* Админ видит статусы тикетов (New, In Progress, Resolved) и может в один клик перейти к редактированию проблемной аптеки.
* В целевом Supabase-состоянии обычный пользователь может только создать жалобу через ограниченную RLS-политику или серверный RPC. Чтение очереди жалоб и смена статусов доступны только администраторам.

### 4.2 Черный список препаратов (Группы риска)
* Админ ведет список ID препаратов (из Эталонной базы), которые система **принудительно скрывает** из поисковой выдачи (например, препараты строгой учетности, лирика и т.д., чтобы избежать проблем с надзорными органами и репутационных рисков).
* Это скрытие происходит на уровне бэкенда, поверх любых других фильтров.
* В целевом Supabase-состоянии управление черным списком выполняется только через админскую роль и защищенную Server Action/RPC. Клиентское приложение не получает прав на изменение этих данных.

## 5. Аналитика Неудовлетворенного Спроса (B2B Lead Generation)

Это важнейший функционал для монетизации и развития сети, упомянутый в бизнес-контексте.

### 5.1 Сбор данных (Zero-Result Searches)
* Если пользователь ищет препарат и система выдает 0 результатов (или пользователь находится далеко от единственной аптеки с этим препаратом), запрос логируется в базу данных.
* Чтобы в отчет не попадали промежуточные буквы live-search, логируется только стабилизированный запрос: минимум 3 символа и пауза ввода не менее 1.5 секунды.
* Логируется: `Поисковый запрос`, `Гео-зона (район/город)`, `Время`.

### 5.2 Дашборд "Дефицитные позиции"
* Админ-панель агрегирует эти данные и выводит отчет: *"Топ-50 препаратов, которые искали, но не нашли в Гудермесе за последнюю неделю"*.
* Система автоматически фильтрует опечатки (используя движок поиска), объединяя запросы "аспирин" и "асперен" в единый счетчик.

### 5.3 Использование для бизнеса (Ценность для аптек)
* **Future:** Администратор может выгрузить этот отчет в PDF/Excel и отправить владельцам аптек в данном районе (или показать им лично). Текущий MVP показывает отчет в защищенной админке без экспорта.
* **Смысл:** "В вашем районе 150 человек за неделю искали препарат X. Закупите его, загрузите нам прайс, и все эти клиенты придут к вам". Это превращает платформу из простого справочника в инструмент генерации выручки для аптек.
